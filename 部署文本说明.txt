3. 检查你的根目录结构  确保最终完整结构如下：
.docker（文件夹）
plugins（文件夹）
storage（文件夹）
theme（文件夹）
.env（配置信息文件）
compose.yaml（docker compose文件）


4. 打开.env配置信息文件，填写自己域名，填写自己本地Redis信息（镜像不包Redis、NGINX、PostgreSQL）
还需要填写compose.yaml内的Redis、PostgreSQL的容器名称

5. 给根目录下的storage文件夹755权限

6. 数据库表字段类型修复，修复对postgresql数据库的支持！（不执行修复你xboard运行日志会一直有报错日志）修复脚本见postgresql_types.sh

7. 添加站点设置反向代理和NGINX信息（如果前面你第一步就已经创建并且配置了证书跟HTTPS，则这里只需要修改一下NGINX配置信息）
见xboard.conf
确保你服务器放行你自己映射的端口、7001、7002、8076、包括后续你添加节点的端口号。


8. 最后执行启动命令
docker compose up -d
作用： 在后台构建、（重新）创建、启动并附加到容器。
场景： 第一次启动服务，或者修改了 compose.yaml / .env 文件后应用新配置。


停止运行命令
docker compose down
作用： 停止容器，并删除容器、网络、数据卷（未声明外部或持久化的）和镜像。
场景： 彻底关闭并清理当前服务环境。(注：若要连同挂载的数据卷一起删除，加 -v 参数)。


docker compose stop
作用： 仅仅停止正在运行的容器，不删除它们。
场景： 临时暂停服务，稍后还会原样启动。


docker compose start
作用： 启动已经存在但处于停止状态的容器。
场景： 配合 stop 命令使用。


docker compose restart
作用： 重启容器。
场景： 服务出现卡顿，或者修改了容器内部不涉及 yaml 结构的文件（如某些代码热更新失败时）需要硬重启。
特例： docker compose restart <服务名> 可以只重启特定的服务（例如只重启 xboard，不重启数据库）。




查看状态与日志 (监控与排错)
docker compose ps
作用： 列出当前 Compose 项目下的所有容器及其状态（运行中、退出、端口映射等）。
场景： 检查服务是否都在正常运行（Up 状态）。加 -a 参数可查看包含已停止的容器。

docker compose logs
作用： 查看服务的控制台日志。

常用组合：
docker compose logs -f
实时滚动查看日志（按 Ctrl+C 退出）。

docker compose logs --tail=100
只看最后 100 行日志。

docker compose logs -f <服务名>
只看某个特定服务（如 xboard）的实时日志。


docker compose top
作用： 显示各个容器内正在运行的进程。
场景： 检查容器内是哪个进程占用了过高的 CPU 或内存。




更新镜像与构建
docker compose pull
作用： 拉取 compose.yaml 中定义的所有镜像的最新版本（基于 tag，比如 latest 或 0.0.1）。
场景： 远程镜像仓库更新了，需要同步到本地服务器。

docker compose build
作用： 根据 Dockerfile 重新构建镜像。
常用组合： docker compose build --no-cache （不使用缓存，强制完全重新构建）。

一键更新服务组合拳：
Bash
docker compose pull
docker compose up -d
说明：这两步会自动拉取新镜像，并仅重建那些镜像发生了变化的服务容器，其他服务不受影响，是平滑升级的最佳实践。



进阶与清理命令
docker compose config
作用： 验证并查看最终解析后的 Compose 配置。
场景： 检查环境变量（.env）是否正确注入，或者 yaml 语法是否有错。

docker compose rm
作用： 删除已停止的容器。

全局 Docker 清理（非 Compose 独占，但极常用）：
docker system prune -a
清理所有未使用的数据（停止的容器、未使用的网络、悬空镜像）。⚠️ 生产环境慎用，会删掉所有未在运行的镜像。

docker image prune -f
只清理没有被任何容器使用的旧/废弃镜像（构建多次后清理磁盘空间的利器）。
